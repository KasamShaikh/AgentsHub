Microsoft Agent Framework edition — architecture, agents, flow & open-source options

ⓘ What this app is

An agentic email-processing pipeline on Azure. Emails land in an Office 365 inbox; an orchestrator agent reads each one, decides what kind of email it is, and routes it to a specialist agent that does the real work — extracting trades from a broker contract note, or cross-checking a merchant’s onboarding forms — and writes a clean, auditable output file back to storage.

Version 2 · Microsoft Agent Framework (MAF). Functionally identical to v1, but the agent runtime is now the open-source Microsoft Agent Framework running in-process instead of the Azure AI Agents SDK and the Foundry-hosted agent runtime. The model (gpt-mini-ks), Logic App ingestion, Document Intelligence, Blob storage and every output file are unchanged. Agents are now defined in code and their tools are invoked automatically by the framework — there is no server-side threads/runs object and no manual tool-output plumbing. This is a separate copy of the app; the original v1 deployment and its Azure resources are left exactly as they are.

▤ High-level architecture

Email in → Logic App stages attachments to Blob → the in-process Microsoft Agent Framework orchestrator (gpt-mini-ks) classifies the intent and auto-invokes its tool → code routes to a specialist → the result is written back to Blob. Every call uses Entra managed identity — no keys.

🧠 The model & platform

LLM: a GPT‑class small model deployed on Azure AI Foundry (deployment gpt-mini-ks), chosen for low latency and cost on high-volume mailbox traffic. Unchanged from v1.

Agent runtime (v2): the open-source Microsoft Agent Framework. Agents are defined in code as in-process ChatAgent objects with their own instructions and tools; the framework invokes a tool automatically and feeds the result back — no Foundry-hosted threads/runs and no manual tool-output loop. The model is still served by Foundry gpt-mini-ks.

OCR: Azure AI Document Intelligence (prebuilt-layout, markdown output) turns scanned PDFs and images into text the agents can read.

Auth: Microsoft Entra managed identity only — no keys. Blob shared-key access is disabled.

⚛ The core design principle

The LLM proposes; deterministic code decides.
Agents do the fuzzy, language-heavy work — reading documents, aligning fields that are worded differently. Plain Python then makes the final, repeatable decision (scoring, formatting, thresholds).

Example: in onboarding, form-compare-ks aligns the two forms semantically, but a deterministic scorer applies fixed thresholds (≥0.95 = match, ≥0.80 = near, else mismatch) so the same forms always yield the same verdict — essential for an auditable process.

➤ How a message flows, end to end

  1. Email arrives A new mail hits the Office 365 inbox and triggers the logic-email-ks Logic App.
  2. Ingest & stage The Logic App saves each attachment to the incoming-attachments blob container and composes a JSON payload (subject, body, blob paths).
  3. Classify intent The in-process Microsoft Agent Framework orchestrator (gpt-mini-ks) reads the payload and classifies it: contract_note, pre_onboarding or manual. For a contract note it auto-invokes its run_contract_pipeline tool — the framework runs the real pipeline and feeds the summary back, with no manual tool-output step.
  4. Route to a specialist Code-driven routing hands the work to the matching in-process agent — the model decides the intent, the app decides the wiring.
  5. Do the work  • Contract note: OCR → extract trades → resolve identifiers → build the pipe-delimited file.
     • Onboarding (demo): OCR both forms → agent aligns fields → deterministic scorer computes match % and flags mismatches.
  6. Write output The result lands in the contract-notes-output container (a .txt report + machine .json).
  7. Observe Every step is streamed to the live dashboard over Server-Sent Events and saved as a replayable trace.

🗂 Inputs, outputs & storage

Input: incoming-attachments — raw email attachments.

Demo forms: onboarding — the two fixed sample forms used for the onboarding comparison.

Output: contract-notes-output — generated upload files and onboarding reports.

All access is via managed identity; nothing is read from or written with account keys.
